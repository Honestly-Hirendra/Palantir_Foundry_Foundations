Showcase Script

🎯 Opening Hook ~30 sec

"I recently completed six Palantir Foundry courses — and what I want to emphasize is that I didn't just study the platform, I actually built on it. I built data pipelines, wrote PySpark transforms, designed an Ontology, created a production-grade Workshop application, and topped it off by shipping an AI-powered Knowledge Graph app. Today I'd like to walk you through what that journey looked like and why I think it maps directly to what you're trying to solve."

TIP: Pause after "I actually built on it." Let that land. Eye contact here is crucial.

🔧 Courses 1–3 · The Data Layer — Contour → Pipeline Builder → Code Repositories ~75 sec

"My first course was Contour — Foundry's data analysis and visualization tool. It organizes analysis into linear dependent paths of logic blocks, which keeps things interpretable and auditable. From there I moved to Pipeline Builder, where I built a fully functional end-to-end data pipeline with no code at all — branching, scheduling, validation, all handled through a visual interface backed by Spark. Then I went deeper with Code Repositories, writing actual PySpark — joins, aggregations, casting, filtering — in Foundry's browser-based IDE, with collaborative branching and Data Lineage integration for full governance. I now understand all three ways data gets transformed in Foundry: visual analysis in Contour, low-code pipelines in Pipeline Builder, and production-grade code in Code Repositories."

TIP: Draw three boxes left to right as you speak: Contour → Pipeline Builder → Code Repos. Then say: "Each one goes deeper — ad-hoc analysis, to no-code pipelines, to full PySpark." That progression signals architectural maturity, not just tool familiarity.

🗂️ Course 4 · The Semantic Layer — Ontology Deep Dive ~60 sec

"Once the data is clean, Foundry makes it meaningful through the Ontology — and this is where things get interesting. The Ontology isn't just a data catalog or a semantic label. It encodes the nouns AND the verbs of your organization. The nouns are your object types — Aircraft, Shipment, Patient — and the verbs are actions that enforce business rules and write decisions back into the platform. I defined my own object types, relationships between them, and actions — and I learned how this single layer powers every downstream tool: Workshop, AIP, Quiver, and the Ontology SDK."

TIP: Say "nouns AND verbs" clearly and slowly. This is the insight that separates people who really understand Foundry from those who just used it.

📦 Course 5 · The Application Layer — Workshop + Shipment Ops Center ~60 sec

"With a solid Ontology in place, I built a real operational application in Workshop — the Shipment Operations Center. This isn't a prototype — it's an app that shipment managers can actually use to understand what's happening across their supply chain and take action on it. I built modules from scratch, configured and connected widgets through variables, managed layouts with sections, applied conditional formatting, and defined actions — both simple point-and-click and complex function-based actions. Crucially, every action a user takes is written back into the Ontology, so all downstream workflows and users benefit from that decision."

TIP: If you have a screen recording or screenshot of the app, show it here. Visuals convert abstract descriptions into concrete proof.

🤖 Course 6 · Capstone — AIP Logic + Knowledge Graph App ~60 sec

"The capstone of my training was building an end-to-end AI workflow using AIP Logic. Starting from a set of PDFs, I built a transformation pipeline that chunked the documents, used an LLM to extract the most important entities from each chunk in batch, and loaded those chunks and entities into the Ontology as objects. This created a Knowledge Graph — a structured semantic map of the research domain. I then configured an AIP Agent that uses this graph to answer natural language questions reliably, grounded in trusted data — no hallucinations, because every answer traces back to a real source in the Knowledge Graph. Finally, I surfaced the whole thing in a Workshop app so any user can query the knowledge base and see exactly which nodes of the graph were used to formulate the answer."

TIP: This is your strongest card. Slow down here. Say "no hallucinations, because every answer traces back to a real source." That phrase will resonate with anyone who's deployed LLMs in enterprise settings.

🔗 The Architecture Summary — The Closing Synthesis ~30 sec

"What I want you to take away is that I don't just know individual Foundry tools — I understand how they compose. Data moves from raw sources through Pipeline Builder and Code Repositories, gets shaped by Contour, becomes meaningful through the Ontology, powers operational applications in Workshop, and serves as the trusted foundation for AI through AIP. Each layer depends on the one below it. I can reason about any point in that stack, and I can communicate across it — between data engineers, ontology designers, app builders, and AI teams."

TIP: Draw the stack as you say it. Literal stack — five layers. This is the moment you look like an architect, not a student.

How to Stand Out

01

Draw the Stack Live

Most candidates talk about Foundry. You should draw it — a five-layer diagram (Raw Data → Pipelines → Ontology → Workshop → AIP) sketched in real time shows systems thinking, not memorization. No one forgets the person who drew the architecture.

02

Own the Ontology Insight

Drop this line naturally: "The Ontology isn't just a semantic layer — it's also where logic and kinetics live. It's the nouns AND the verbs." Most candidates who've done the course miss this nuance. Landing it shows depth of understanding, not surface familiarity.

03

Lead With the AI App

The Knowledge Graph + AIP Logic app is your peak achievement. Describe the architecture in detail: PDF → chunk → LLM batch extraction → Ontology objects → Knowledge Graph → AIP Agent → Workshop UI. Anyone familiar with RAG pipelines will recognize this as production-grade design.

04

Show a Screen Recording

Record a 90-second demo of the Shipment Ops Center or the TB Q&A app in action. Showing a real working interface shifts you from "someone who did a course" to "someone who built something." This is the single highest-ROI thing you can prepare.

05

Bridge to Their Domain

Before the interview, research what the organization does. Then say: "The same Ontology + AIP pattern I used for Tuberculosis research — I can immediately see how that applies to [their domain]." Contextualizing your work to their problem is the fastest way to be remembered.

06

Speak the Trade-offs

When asked about tools, don't just list them. Say: "Contour is ideal for ad-hoc analysis and dashboards, but I'd choose Code Repositories for high-scale pipelines needing governance and incremental compute." Knowing WHEN to use each tool signals real practitioner judgment, not just course completion.
